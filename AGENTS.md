# Prototype Instructions

Run the local server yourself and open the preview in the in-app browser. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

When the user provides interior photos and asks for a new room/background preset, use the `sofa-background-preset` skill at `/Users/dima/.codex/skills/sofa-background-preset/SKILL.md`. The user should not need to repeat the workflow; create or clean UGC/iPhone room references, save project assets, wire the UI card and server scene prompt, preserve product-vs-room reference separation, and validate locally.

For new chats or architecture handoff, read `ARCHITECTURE.md` first. It documents the current frontend/server/shared layout, generation modes, shot sets, room presets, prompt locks, fabric replacement flow, validation commands, and known generation risks.

## Selected product direction

- The approved source visual is `/Users/dima/Documents/Снимок экрана 2026-06-23 в 01.25.40.png`.
- Preserve the light editorial split-screen composition, warm ivory surfaces, high-contrast serif display type, restrained oxblood accent, and large photorealistic sofa image.
- Keep the core flow on one calm screen: 1-6 references, compact room preset selection, 3/4 outputs, generation, results, and API cost.
- Fabric replacement belongs inside the existing results section: six visual material presets, one synchronous hero preview, then immediate or 50%-discount Batch processing for the remaining shots.
- Fabric edits must always start from the original generated shot, never from an already edited fabric variant. Preserve sofa geometry, seams, non-upholstered parts, room, lighting and camera.
- Built-in Codex image generation may create static preset assets without consuming the product API key; runtime customer edits use the configured OpenAI API key.
- Background expansion should start with admin-curated presets, not user-uploaded room backgrounds. Each preset should pair a UI card, a scene prompt, and internal room reference images generated or cleaned from admin-supplied references. For the first UGC preset, use the recognizable compact apartment with window, taupe curtains, ivory ribbed rug, pale oak floor, arched floor lamp, abstract wall art, and organic ceiling light; keep it realistic smartphone/UGC rather than catalogue or CGI.
- Background reference assets should avoid sterile render-clean output. Prefer a subtle iPhone/UGC treatment for admin presets: gentle highlight rolloff, mild phone/JPEG texture, slightly imperfect exposure and white balance, and softened CGI-sharpness while preserving room geometry and placement space.
- The `UGC гостиная` preset uses the gray-beige herringbone living room references: limewashed plaster wall, white baseboards, pale-oak herringbone floor, ivory patterned rug, taupe curtains at the left window, slim brass floor lamp, and two framed abstract posters. Place the sofa along the main wall under the posters, grounded on the rug, while keeping the lamp/window context readable.
- New background presets should be validated with internal QA storyboards before handoff: generate/check the 4-shot set (`Hero`, `Front`, `Depth`, `Detail`) and, when the preset is full-ready, the 6-shot set (`Hero`, `Front`, `Depth`, `Detail`, `Elevated`, `Room`) across multiple sofa archetypes. Treat these QA storyboards as admin validation assets, not final customer exports.
- Replace raw output count selection with named shot sets: a 4-shot quick set and a 6-shot full shoot. Collage/contact-sheet generation may be used as a cheaper preview/selection step, but final customer images should be generated as separate high-quality shots from the original references and selected room preset.
- Default finalization should now be preview-locked: after a good contact-sheet preview, use the selected cropped preview frames as composition anchors and the user product references as the sofa identity source. Independent final reshoots remain a secondary, higher-drift fallback.
- For the next product-fidelity benchmark, remove final image generation from the tested workflow. Treat contact-sheet collages and their cropped preview tiles as the test output; preview-lock/finalization is a later separate stage, likely with another model.
- Benchmark variants should start with A1, B1 and C1 across all eight `qa/input-sofas` sofas, plus D1 as a separate product-image-edit branch: A1 = OpenAI contact sheet with a generated Sofa Passport, B1 = Gemini Flash Image contact sheet with a Sofa Passport, C1 = Gemini Pro Image contact sheet with a Sofa Passport, D1 = product-image/background-edit experiment for selected Hero/Front-style cases.
- In the first benchmark version, visual checking may run and write logs/scores, but it must not block generation, trigger retries, or change prompts at runtime. Use it as passive QA evidence only.
- Benchmark reports/PDFs must include the original sofa references next to generated previews. Sofa Passport must encode section geometry beyond counts, especially unequal module widths such as narrow-left / wide-center / narrow-right, and must include no-equalization rules when the source layout is asymmetric.
- Problem sofas with asymmetric section widths can pass in clean interior/chat generations but fail in iPhone/UGC mode by becoming slightly narrower or regularized. Treat phone realism as lower priority than product width: preserve full sofa width, narrow-left / wide-center / narrow-right base ratios, pull-tab positions, seams, mismatched base/back cushion counts, and footprint; never create iPhone realism by squeezing or equalizing the product.
- Sofa legs are a hard product-fidelity acceptance criterion. Preserve leg count, position, material, color, height, thickness and shape; do not invent generic wooden/tapered legs or move the sofa footprint. If a hidden leg is unknowable, keep it occluded or minimally inferred.
- Chat/Codex image generation may be used as a cheap simulation QA layer for prompt sequence, composition, leg fidelity and folder outputs, but it is not production proof. Keep one minimal real API smoke test before trusting runtime behavior.
- OpenAI `moderation: "auto"` is only safety moderation. Do not treat it as a second-level product QA system for geometry, legs, footprint or fabric fidelity.

Background preset cards stay visible, but this route is text-only for environments: never send internal room reference images through `/api/generate-frame`.
